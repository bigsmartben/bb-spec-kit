## 需求概述

本需求旨在为 Spec 模板补齐产品设计规范层，解决仅靠工程验收字段难以支撑产品评审、交互走查与状态逻辑校验的问题。通过在 Spec 中引入 Actor/边界、交互流程、状态机、UI 组件规范，并建立与下游 Plan/Tasks 的追溯闭环，降低跨角色歧义与实现偏差。

---

## 背景与问题

当前 `spec-template.md` 面向**工程验收**设计，核心内容为：

- User Scenarios（Given-When-Then 验收场景）
- Functional Requirements（FR-xxx 能力声明）
- Success Criteria（可量化的成功指标）

### 存在的问题

**1. 用例图缺失**

Actor、系统边界、Use Case 关系从未被显式声明，角色信息隐含在自然语言里，评审阶段对"谁能操作什么"存在解读歧义。

**2. 用户交互流程图缺失**

Given-When-Then 是验证语言，不是流程控制语言，无法表达：
- 条件分支（步骤 A 失败后走哪条分支）
- 泳道（多角色交替操作的顺序）
- 系统异步处理期间用户感知到的界面变化

**3. 状态机图缺失**

Edge Cases 零散描述异常状态，但缺少完整的：
- 状态枚举（穷举所有合法状态）
- 转换矩阵（触发事件 × 守卫条件 → 目标状态）
- 禁止转换清单（明确不允许的跳跃路径）

导致开发实现状态逻辑时仍需猜测或自行决策。

**4. UI 元素定义完全缺失**

FR-xxx 只声明系统能力，没有任何 UI 维度，开发者无法获知：
- 字段的**内涵**（业务语义是什么）
- 字段的**口径**（如何取值、格式、边界）
- 组件在不同状态下的展示和交互规则

### 影响范围

| 受影响方 | 问题表现 |
|---------|---------|
| 产品评审 | 评审方用自然语言解读规格，存在解读偏差 |
| 开发实现 | UI 细节由工程师自行决策，标准不统一 |
| 测试验收 | 状态转换覆盖不完整，边界场景遗漏 |

---

## 需求目标

在 spec 阶段新增**产品设计规范层**，服务于：

1. **产品设计与评审**：标准化、无歧义地描述系统行为
2. **下游开发**：提供精确到组件级别的实现契约
3. **测试覆盖**：状态机驱动的完整测试用例生成

---

## 用户场景

### 场景 1：产品评审

产品经理在评审阶段可通过 UC 明细、交互流程与 UI 组件定义快速确认“谁在什么条件下做什么”，并通过 `FR/EC` 引用检查是否覆盖关键业务规则。

### 场景 2：开发实现

开发者基于 Spec 的业务锚点（组件依赖、`Entity.field` 引用）在 Plan 中补齐接口/事件/存储实现映射，避免在需求阶段过早绑定技术细节。

### 场景 3：测试验收

测试人员可从 Spec 的验收场景和状态机规则生成测试路径，并在 Tasks 阶段验证实现任务是否可追溯到 `UC/FR` 与 `Entity.field`。

## 功能需求

### FR-001：新增"参与者与系统边界"小节（用例图文本表达）

**目标**：显式声明所有参与者（Actor）和系统边界。

**内容要求**：
- 列举所有 Actor（人类用户、外部系统、定时任务）及其权限与职责
- 声明本 Spec 的系统边界（覆盖什么，不覆盖什么）
- 描述 Use Case 之间的关系（`<<include>>` / `<<extend>>`）
- Use Case ID 与 User Story ID 双向引用

**属性**：可选小节，`*(optional)*` 标注

---

### FR-002：新增"用户交互流程"小节

**目标**：以结构化流程语言表达用户与系统的交互过程。

**内容要求**（参考 `.spec-workspace/user_template/交互流程图.md`）：

每条流程包含：
- **前置条件**：进入此流程时，系统与用户的已知状态
- **主流程**：Actor 操作 → 系统响应，逐步编号（支持 mermaid 流程图）
- **分支点**：条件判断，说明满足/不满足时的走向
- **异常路径**：每个可能失败节点的处理路径（含用户可感知的界面反馈）
- **后置条件**：流程结束后的系统状态

**与现有内容的关系**：
- 流程中每个系统节点 → 标注对应 `FR-xxx`
- 流程中每条异常路径 → 标注对应 `EC-xxx`（Edge Cases）
- 通过 ID 交叉引用，不重复描述内容

**属性**：可选小节，`*(optional)*` 标注

---

### FR-003：新增"实体状态机"小节（状态机图文本表达）

**目标**：对有生命周期的关键实体，集中管理所有状态及转换规则。

**内容要求**：

每个状态机包含：
- **状态枚举**：所有合法状态及其业务语义定义（必须穷举）
- **初始状态 / 终止状态**：明确标注
- **转换矩阵**：`From State × Event → To State`，附带守卫条件（Guard）
- **禁止转换清单**：明确不允许的转换（等同于状态类 Edge Cases）
- **并发说明**：同一实体是否可能被并发触发，以及冲突解决规则

**与现有内容的关系**：
- `Key Entities` 描述实体**属性** → State Machine 描述实体**行为时序**，通过实体名称关联
- `Edge Cases` 中状态类条目 → 迁移至此处的"禁止转换清单"
- `Edge Cases` 保留环境类异常（网络、权限、并发超时）

**属性**：可选小节，`*(optional)*`，注明"仅当涉及有生命周期的实体时填写"

---

### FR-004：新增"UI 元素定义"小节

**目标**：提供功能点级别的前端实现契约，精确定义每个 UI 组件的内涵与口径。

**内容要求**：

组织方式：**按 User Story 分组**，每组下列出所有相关组件。

每个组件条目包含：
- **组件标识符**（唯一 ID）
- **类型**（Button / Input / Select / Table / Modal / Badge 等）
- **显示文案**（精确到字符，含各状态文案）
- **内涵**（该字段/控件的业务语义，**必填**）
- **口径**（取值规则、数据来源、格式约束、边界值，**必填**）
- **状态定义**（`enabled` / `disabled` / `loading` / `error` / `empty` 各状态规则）
- **触发行为**（用户操作后发生什么，关联 `FR-xxx`）

> 注：Spec 阶段不要求填写接口端点、请求/响应字段、存储映射等实现细节；
> 这类内容在下游 plan/design 阶段补充。

**"内涵与口径"强制性**：两项必填，不允许留空，是消除歧义的核心。

**与现有内容的关系**：
- `Key Entities` 作为数据层单一真相，UI 层通过字段 ID 引用，不重复定义口径
- FR-xxx 是能力声明，UI 组件的"触发行为"字段通过 FR ID 引用

**属性**：可选小节，`*(required for frontend features)*`，前端功能强制填写

---

## 非功能需求

- **向后兼容**：新增小节均为可选，不破坏现有 `spec-template.md` 结构，所有已有 Spec 文件不受影响
- **最小化冗余**：通过 ID 交叉引用避免内容重复
- **可读性**：每个小节有清晰的 `*(optional)*` / `*(required)*` 标注

## 闭环补充（2026-03-03）

为落实“Spec 保语义，Plan 承实现”的决策，已在实验工作区新增闭环模板：

- `.spec-workspace/templates/plan-template-EXT-002.md`
- `.spec-workspace/templates/tasks-template-EXT-002.md`
- `.spec-workspace/templates/commands/plan-EXT-002.md`
- `.spec-workspace/templates/commands/tasks-EXT-002.md`

闭环规则：

1. Spec 侧保留 `Key Entities` 与 UI 语义锚点（内涵/口径/状态规则/触发行为）；
2. Plan 侧补齐 `Interface & Implementation Mapping`（接口/事件/存储等实现细节）；
3. Tasks 侧同时消费 Spec 锚点与 Plan 映射，要求每条实现任务可追溯到 `Entity.field`。
