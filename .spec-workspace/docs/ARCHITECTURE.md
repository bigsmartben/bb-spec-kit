# Spec Kit 架构决策记录 (ADR)

> 📖 **目的**: 记录重要的架构决策及其理由

## 什么是 ADR?

Architecture Decision Record (架构决策记录) 是记录重要架构决策的轻量级文档。每个 ADR 描述一个决策的背景、决策内容和后果。

---

## ADR 模板

每个 ADR 应包含：

1. **编号**: 唯一标识符 (ADR-XXX)
2. **日期**: 决策制定日期
3. **状态**: 提议/已接受/已废弃/已替代
4. **背景**: 为什么需要做这个决策
5. **决策**: 决定做什么
6. **后果**: 这个决策的影响（正面和负面）
7. **相关决策**: 与其他 ADR 的关系

---

## ADR-001: 创建独立的开发工作区

**日期**: 2026-03-03  
**状态**: 已接受  
**决策者**: Spec Kit 开发团队

### 背景

在扩展 Spec Kit 时，开发者直接修改主仓库文件存在以下问题：

1. **污染风险**: 实验性修改可能影响主仓库的稳定性
2. **追溯困难**: 难以追踪每个需求的完整开发过程
3. **协作混乱**: 多人同时开发时容易产生冲突
4. **规范缺失**: 没有统一的需求管理和验证流程

### 决策

创建 `.spec-workspace/` 独立工作区，包含：

- `requirements/` - 需求管理和文档
- `templates/` - 实验性模板
- `sandbox/` - 测试和实验
- `docs/` - 开发文档
- `tools/` - 辅助工具

**原则**：
1. 所有扩展开发先在工作区进行
2. 通过验证后才能同步到主仓库
3. 完整记录需求、设计、实现、测试全过程
4. 使用工具脚本控制同步流程

### 后果

**正面影响**：
- ✅ 主仓库保持稳定，不受实验影响
- ✅ 完整的开发过程追溯
- ✅ 规范化的工作流程
- ✅ 更好的团队协作
- ✅ 降低引入 bug 的风险

**负面影响**：
- ❌ 增加了一层抽象，学习曲线略陡
- ❌ 需要维护工作区和主仓库两套文件
- ❌ 同步过程需要额外步骤

**缓解措施**：
- 提供清晰的 README 和 WORKFLOW 文档
- 开发辅助工具脚本自动化常见操作
- 在文档中提供详细示例

## ADR-002: 使用 Markdown 作为主要文档格式

**日期**: 2026-03-03  
**状态**: 已接受  
**决策者**: Spec Kit 开发团队

### 背景

需要选择一种文档格式来记录需求、设计、测试等信息候选方案：
- Markdown
- reStructuredText
- AsciiDoc
- YAML
- JSON

### 决策

使用 Markdown 作为主要文档格式。

**原因**：
1. 与 Spec Kit 核心格式一致
2. 易读易写，学习成本低
3. 广泛的工具支持（编辑器、预览、转换）
4. GitHub 原生支持
5. 可以包含代码块、表格、列表等丰富元素
6. 纯文本，友好的 Git diff

### 后果

**正面影响**：
- ✅ 文档易于编写和维护
- ✅ 与项目其他文档风格一致
- ✅ 良好的可读性
- ✅ 工具支持广泛

**负面影响**：
- ❌ 缺少严格的结构验证（相比 YAML/JSON）
- ❌ 不适合存储结构化数据

**缓解措施**：
- 使用验证脚本检查文档完整性
- 为重要章节使用明确的标题标记
- 提供模板确保一致性

## ADR-003: 按需求ID组织文件结构

**日期**: 2026-03-03  
**状态**: 已接受  
**决策者**: Spec Kit 开发团队

### 背景

需要决定如何组织需求相关文件：

**方案A**: 按文件类型组织
```
requirements/
├── requirement-docs/
├── designs/
├── implementations/
└── tests/
```

**方案B**: 按需求ID组织
```
requirements/
├── EXT-001/
│   ├── requirement.md
│   ├── design.md
│   ├── implementation.md
│   └── tests.md
└── EXT-002/
    └── ...
```

### 决策

采用方案B：按需求ID组织文件。

**理由**：
1. 每个需求的所有文档集中在一起
2. 更容易管理和查找
3. 可以完整复制/移动整个需求
4. 清晰的所有权和追溯性

### 后果

**正面影响**：
- ✅ 文档组织清晰
- ✅ 需求的完整性一目了然
- ✅ 便于归档和查找
- ✅ Git 历史更清晰

**负面影响**：
- ❌ 跨需求的文件比较略麻烦（可通过工具缓解）

## ADR-004: 工具脚本使用 Bash

**日期**: 2026-03-03  
**状态**: 已接受  
**决策者**: Spec Kit 开发团队

### 背景

需要选择工具脚本的实现语言：
- Bash
- Python
- Node.js
- PowerShell

### 决策

使用 Bash 实现工具脚本。

**原因**：
1. 与项目现有脚本一致 (`scripts/bash/`)
2. Linux/macOS 原生支持
3. 适合文件操作和命令编排
4. 无需额外依赖
5. 团队熟悉度高

### 后果

**正面影响**：
- ✅ 与项目技术栈一致
- ✅ 无额外安装要求
- ✅ 性能良好

**负面影响**：
- ❌ Windows 用户需要 WSL 或 Git Bash
- ❌ 复杂逻辑实现困难

**缓解措施**：
- 文档中说明 Windows 使用方式
- 保持脚本逻辑简单
- 未来可考虑添加 PowerShell 版本

## ADR-005: 状态驱动的工作流

**日期**: 2026-03-03  
**状态**: 已接受  
**决策者**: Spec Kit 开发团队

### 背景

需要定义清晰的需求状态流转机制。

### 决策

采用明确的状态定义和转换规则：

**状态定义**：
- 待评审 → 设计中 → 开发中 → 测试中 → 已完成 / 已拒绝

**转换规则**：
- 每个状态有明确的准入和产出标准
- 状态转换需要完成对应文档
- 使用验证工具检查状态转换条件

### 后果

**正面影响**：
- ✅ 清晰的开发流程
- ✅ 可追溯的进度
- ✅ 明确的质量门禁
- ✅ 便于团队协作

**负面影响**：
- ❌ 流程可能显得繁琐（小需求）

**缓解措施**：
- 允许简单需求快速通道
- 工具自动化常见检查
- 保持流程灵活性

## ADR-006: 集中式需求管理表

**日期**: 2026-03-03  
**状态**: 已接受  
**决策者**: Spec Kit 开发团队

### 背景

需要决定如何管理需求列表：

**方案A**: 分散在各需求目录中  
**方案B**: 集中在 REQUIREMENTS.md  
**方案C**: 使用 GitHub Issues

### 决策

采用方案B：集中式 REQUIREMENTS.md + 详细文档在需求目录。

**理由**：
1. 快速浏览所有需求
2. 便于状态追踪和看板展示
3. 与 Git 工作流集成良好
4. 不依赖外部系统

### 后果

**正面影响**：
- ✅ 一站式需求概览
- ✅ 易于搜索和过滤
- ✅ 纯文本，Git 友好

**负面影响**：
- ❌ 大型表格维护略繁琐
- ❌ 缺少 Issue 系统的互动功能

**未来考虑**：
- 可选择性地创建 GitHub Issues 链接
- 开发脚本辅助表格维护
